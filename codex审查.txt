  ### Round 3 新发现（5 项）

  1. [P1] U14 — ApiKeyAuthFilter 用 Spring WebFlux WebFilter，但项目架构是 Spring MVC

  - 审阅者： feasibility-reviewer (置信度: 75), coherence-reviewer (置信度: 75)
  - 部分： Implementation Units → U14
  - 类型： error
  - autofix_class: gated_auto
  - 重要性： U14 Approach 说 ApiKeyAuthFilter（Spring WebFlux WebFilter）拦截 /api/**，但整个项目使用的是标准 Spring
    Boot MVC 栈（Spring AI ChatClient、@EnableAgentFlow AutoConfiguration、TraceController/DiagnosisController 等都是标
    准 @RestController）。WebFlux 的 WebFilter 在 Spring MVC 应用中不生效——MVC 用的是 OncePerRequestFilter 或
    HandlerInterceptor。如果实现者照做，鉴权过滤器将完全不工作。

  - 证据： ApiKeyAuthFilter（Spring WebFlux WebFilter）拦截所有 /api/** 请求 (U14 Approach) ... 无其他 WebFlux 标识
  - 建议修复： 将 ApiKeyAuthFilter 改为继承 OncePerRequestFilter（Spring MVC 标准）或使用 HandlerInterceptor。移
    除"Spring WebFlux WebFilter"措辞。

  2. [P1] U14 — 返回 202 Accepted 但 Approach 说"v1 同步"执行

  - 审阅者： coherence-reviewer (置信度: 75)
  - 部分： Implementation Units → U14 → POST /workflows
  - 类型： error
  - autofix_class: manual
  - 重要性： U14 说 WorkflowController 接收 YAML/inputs → 调用 BspEngine.execute（v1 同步）→ 返回 202 + Location。但
    HTTP 202 表示"请求已接受、异步处理中"——如果 v1 是同步执行，应返回 200 OK + 响应体（执行结果）。返回 202 但实际同步阻
    塞等待，会让调用方误以为可以轮询 status 端点，而实际请求已经同步完成了。

  - 证据： 调用 BspEngine.execute（v1 同步）→ 返回 202 + Location: /api/workflows/{id}/status (U14 Approach) ... GET /
    workflows/{id}/status 状态查询 (Files)

  - 建议修复： 同步执行应返回 200 OK + 结果 JSON；或如果确实要返回 202，需说明 BspEngine.execute 是 @Async 异步提交（但
    这与"v1 同步"矛盾）。

  3. [P1] U14 — 测试文件路径在错误模块

  - 审阅者： coherence-reviewer (置信度: 75)
  - 部分： Implementation Units → U14 → Files
  - 类型： error
  - autofix_class: safe_auto
  - 重要性： U14 的 ApiKeyAuthFilter 和 WorkflowOwnershipChecker 源文件在 agentflow-api 模块下（agentflow-api/src/main/
    java/com/agentflow/api/security/），但对应的测试文件路径写的是 agentflow-core/src/test/java/com/agentflow/api/
    security/。测试文件应与源文件在同一模块的 src/test 下，否则 Maven 的 surefire 插件不会执行它们。

  - 证据： agentflow-api/src/main/java/com/agentflow/api/security/ApiKeyAuthFilter.java (源) vs agentflow-core/src/test/
    java/com/agentflow/api/security/ApiKeyAuthFilterTest.java (测试)

  - 建议修复： 将测试文件路径改为 agentflow-api/src/test/java/com/agentflow/api/security/ApiKeyAuthFilterTest.java 和
    agentflow-api/src/test/java/com/agentflow/api/security/WorkflowOwnershipCheckerTest.java。

  4. [P2] U14 — 排在 U13 之后但 Week 5-6，Phase 归属不清

  - 审阅者： scope-guardian-reviewer (置信度: 50)
  - 部分： Implementation Units
  - 类型： error
  - autofix_class: gated_auto
  - 重要性： U14 标注 Week 5-6，但物理位置排在 U13（Week 10）之后、Open Questions 之前。Phase 1 是 Week 1-5，Phase 2 是
    Week 5-7。U14 的 Week 5-6 跨越 Phase 1 和 Phase 2 边界，但没有明确归入哪个 Phase。这会让实现者不清楚 U14 应该在
    Phase 1 结束时开始还是 Phase 2 中开始。

  - 证据： U14. API 鉴权 + 凭证管理 + 提交 Controller（Week 5-6） ... ### Phase 2: 调试体验、可观测性与交付（Week 5-
    7） ... U14 排在 #### U13（Week 10）之后

  - 建议修复： 将 U14 移到 Phase 2 区域内（U6 之后），或在 U14 标题中标注归属 Phase。

  5. [P2] BSP mermaid 图仍显示 5 个 super-step（0-4），与 U12 的 4 super-step 不一致

  - 审阅者： coherence-reviewer (置信度: 75)
  - 部分： High-Level Technical Design → BSP 执行模型与分层算法（mermaid 图）
  - 类型： error
  - autofix_class: gated_auto
  - 重要性： U12（投资分析 Demo）描述为"4 super-step（0-based：step 0/1/2/3）"，但 BSP 分层算法区域的示例 mermaid 拓扑图
    （A→{B,C}→D→{E,F}→G）仍显示 5 个 super-step（step 0-4）。这个拓扑恰好与 U12 的拓扑结构相似（混合 fork-join），但不完
    全相同——mermaid 图是算法说明用的通用示例。不过 v4 修订要点说"U12 super-step 数 5→4 修正"，如果实现者将 mermaid 图当
    作 U12 的拓扑参考，会产生混淆。

  - 证据： mermaid 图含 subgraph "super-step 0" 到 subgraph "super-step 4"（5 层）vs U12 描述"4 super-step（step
    0/1/2/3）"

  - 建议修复： mermaid 示例图是通用算法说明，可以保留 5 层——但应明确标注"此图是分层算法的通用示例，非任何 Demo 的具体拓
    扑"，或添加一个 4 层的 U12 专属拓扑图。
